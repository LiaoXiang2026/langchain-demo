# 项目方向

## 是什么

把这个 LangChain Agent + FastAPI + RAG 项目从"通用文档问答演示"，聚焦成**个人微信公众号文章知识库**。用户上传自己写过的微信文章 HTML，Agent 在多篇文章中**归纳用户的观点和立场**，并以"综合 + 引用"形式回答。

## 为什么

- 用户是写公众号的，存量内容在 200 篇以内
- 已经写过的东西，想通过"问"而不是"翻"来二次利用
- 真正的需求是**跨文章挖观点**，不是单篇问答、也不是找链接

## 服务对象

只有用户本人。无需多租户、无需鉴权、无需分享。

## 关键决策（已与用户确认）

| 决策 | 选择 | 影响 |
|------|------|------|
| Q&A 类型 | "挖观点"型 | 跨文档综合 + 引用 |
| 回答形式 | B — 综合 + 引用（出处标注） | prompt 必须强制 LLM 引用具体文章 |
| 上传方式 | 网页上传（与现有 UI 一致） | 复用现有上传组件 |
| 文档来源 | 微信公众号导出的 HTML | 需专门清洗策略 |
| 数据规模 | 最多 200 篇 | 中等量，嵌入成本可接受 |
| 入库方式 | 持续追加（非一次性） | 必须有去重机制 |

## 文件组织

用户的数据形态：
- 顶层文件夹 → 多个子文件夹
- 子文件夹里是 HTML 文章
- 同目录会有 CSS 等噪声文件，需过滤

## 与现有 RAG 的关系

现有 RAG（`src/rag/*`）已实现 TXT/MD/PDF/Word/Excel 的加载和向量检索。本项目方向是：

- **增量扩展** document_loader 支持 HTML + WeChat 清洗
- **新增** 文件夹批量上传 + 递归扫描 + 文件过滤
- **新增** 增量入库去重机制
- **调整** Agent prompt 偏向"挖观点"任务
- 检索链路（embeddings → vectorstore → knowledge_search tool）保持不变

## 不做的事

- 不做鉴权/多用户
- 不做云存储/对象存储
- 不做其他文档格式（专注 HTML + 微信场景）
- 不做跨平台导出/分享

## How to apply

后续所有 plan、todo、phase 都围绕这个定位展开。当出现"要不要加 X 功能"的分歧时，回看这里：是不是"挖观点"必须的功能？能不能删？
